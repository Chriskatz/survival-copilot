一次問答的流向

1
問題進來 — 手持機 → LoRa mesh → Node (RX) → BLE → basestation.py（通過 ? 前綴）。
2
RAG 檢索 — basestation.py 把問題送 /v1/embeddings 取向量；retriever.py 對 index.json 算 cosine top-k。最高分 < 0.40 → LLM 前直接拒答；否則把命中 chunk 當 context 接到問題前。
3
LLM 生成 — basestation.py → /v1/chat/completions（co-pilot, temp 0.1, /no_think）→ 答案。
4
回覆出去 — clean_reply() 去 <think> 空殼 → chunker 切 ≤200B（不切斷碼位）→ sendText DM → Node (TX) → LoRa mesh → 手持機。

為什麼是這個形狀

兩個 process，用 HTTP 橋接

@meshtastic/js 的 BLE 只能在瀏覽器（Web Bluetooth）；meshtastic-python 的 BLE 在 macOS 才穩。QVAC SDK 又是 Node-only。HTTP 是乾淨的接縫。

純 Python 自寫 retriever

不用 @qvac/rag 的 HyperDB，省掉 Node↔Python 橋；hackathon scope 用 numpy cosine 就夠。

跨語言檢索免標語言

EmbeddingGemma 共享 100+ 語言語意空間，中文問題照樣命中英文 chunk，反之亦然。

RAG 前置拒答

門檻 0.40 在 LLM 之前擋掉離題問題，省 token，並給乾淨的「超出範圍」回覆。
